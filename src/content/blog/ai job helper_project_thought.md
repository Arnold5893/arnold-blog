## 速览
做的什么：一个AI PM岗求职助手，为用户分析和解答求职相关问题

项目背景：求职准备过程中，信息获取效率和信息收益低，存在经验贴分布碎片化、内容重复性与差异性并存、对岗位JD分析不够深入等问题，希望有一个工具在求职准备期间辅助自己完成一些工作（例如信息筛选与获取、简历撰写、JD分析等）
## 整个流程
最初想法是爬取牛客面经数据并存入知识库中，让llm基于“知识库”回答用户的问题。然后我去调研了相关的应用，使用下来发现，这些应用存在简历内容模板化、信息/数据捏造等问题，生成的简历不能直接使用；同时我也意识到，如果只是把自己的项目、个人经历等信息发给模型，让它吐出一份简历，在这个过程中没有发挥主观能动性，没有自己的思考和判断，我觉得这样是有问题的。虽然说可能生成了一版看起来比较美观的简历，但是你和这份简历之间是不熟悉的，甚至你对整个项目的具体思路、逻辑都不清晰。

写简历是一个对自己的能力、项目进行梳理和总结的过程。它本身就需要深度思考，没有人比你更熟悉自己，在撰写过程中可以对自己以及项目有一个更深的认知（甚至还能发现一些项目的问题），如果这一部分完全丢给 ai 去做的话，是对自己不负责任；虽然说ai可以给我们提供一定的便利，但我觉得目前来说，它可能提供的更多是效率上的便利，而不是主观判断上的。
再者，如果只是简历生成和问题答疑的话，这个项目本质就是一个套壳，并且可能我用 Claude 得到的回答质量更好。

所以根据调研，我觉得**产品定位**是让模型引导用户在求职准备过程中如何更好地思考/分析问题、从哪些方面着手准备。比如说，给它一份你的简历和目标岗位JD，他能够分析这个岗位需要哪些能力，并提供要求围绕哪些方面进行表述

所以分为两个分支：分支1是求职咨询，解答ai PM岗位相关的知识；分支2是JD分析，根据用户简历和岗位JD进行分析，拆解JD要求、基于用户的真实背景分析匹配程度等

经过上述的调研和思考，我使用claude对上述想法进行实现。初始版本有一个问题分类器，模型会判断用户的问题属于哪个分支，针对不同的问题会用不同分支的 llm 回答，两个分支都接了相同的知识库和模型，即此时两个分支只是模型的 system prompt 不同

在测试模型回答质量的过程中，我会发现各种问题，然后基于这些问题去不断迭代prompt。

在逐步测试中，我发现知识库起到的作用是有限的，尤其是JD分析，很少能用到面经中的内容，此外，在这个场景下以面经为 chunk 比较大，检索的内容很难有效帮助模型回答用户的问题，有时候反而是噪声

和 Claude 讨论之后，我觉得以问题为 chunk 会更有利于检索，更可能对模型内容生成提供帮助，但如果只是简单地将面经拆解成单个问题，会导致每个问题的上下文缺失，而有些问题只有在特定的语境下才有用。解决办法是给每条问题chunk都提供元数据（问题类型，公司，岗位，面试核心主题、本场面试上下文，以及原始面经链接），通过精细每个chunk的内容来降低上下文缺失带来的问题，且便于检索；另外，本次迭代删除了JD分析分支中的知识库支持，减少噪声；同时，还让 Claude对整个知识库中的面经问题进行分析，得出一个比较完整的结论，作为求职咨询分支系统prompt，帮助模型更好地回答用户的问题。

在后续的测试中，我发现，两个不同的 prompt 可以有效规范分支模型的输出，但若进行**混合意图**提问（输入分别属于两条分支的问题，因为一次只能走一条分支，所以这个问题中的某些部分不是这个分支的职能），在没有相应 prompt 约束的情况下，模型可能会生成一些不符合期望的回答

起初我没有想到太好的解决这个问题的方法，跟Claude 讨论后，它给的办法是通过 Prompt 约束，并输出了一长段禁止 prompt（告诉模型这个不能做，那个不能做），他输出这一堆禁止后，我想到了另一个问题：你不告诉模型该做什么，而是写了很多禁止事项，那么模型会不会因为这些过多的约束prompt，导致不敢生成内容、生成内容的质量一般（并且仅通过写禁止Prompt的方式很难解决所有问题，因为问题没办法穷举）。经过测试发现，果然如此，它的输出显得畏首畏尾，回答像是在凑字数，用户无法获得高质量回复。

这个问题目前的解决办法是，设定模型的**边界**，加了一个边界类prompt，告诉llm只做它分内的事情，超出范围的问题，礼貌地告诉用户不在他的能力范围内（以此避免误导用户的问题）

还有，我有考虑如果给模型加的约束 prompt 太多，会导致它生成的内容重心发生变化。比如它本来的职责是帮助用户进行岗位、简历的分析，但因在测试过程中加了很多 prompt 约束导致它的能力倾斜，记不清自己的**核心定位**。目前这个问题的解决方法是把它的核心职责放在前面（因为模型会更重视前面部分的内容），把约束类的内容放到后面。
